PURPOSE:
A chargeable projectile weapon that ramps damage between a min and max
multiplier based on how long the player holds the trigger. It exists
specifically to demonstrate that the same IWeaponFirable interface can
drive completely different mechanics — when the player presses fire,
this weapon begins a charge; when they release, it shoots. The input
code does not need to know the weapon is chargeable; it just calls
Fire() on press and StopFire() on release like every other weapon.

Code    : private IEnumerator ChargeCoroutine()
Explain : The coroutine that ticks currentChargeTime every frame.
          Private because only StartCharge should launch it.

Code    : while (true)
Explain : Infinite loop — the coroutine runs until it is explicitly
          stopped by ReleaseCharge() or OnDisable(). This is safer than
          an "exit when full" loop because even at full charge the
          player may still be holding the trigger, and the bar should
          stay pinned at 100%.

Code    : currentChargeTime += Time.deltaTime;
Explain : Accumulates charge time using deltaTime, which scales with
          Time.timeScale. That means slow-motion correctly slows the
          charge.

Code    : currentChargeTime = Mathf.Min(currentChargeTime, maxChargeTime);
Explain : Clamps the charge time at the max so the percent computation

Code    : WeaponEventBus.RaiseChargeChanged(GetChargePercent());
Explain : Pushes the current charge percent to any listener each frame.
          Listeners animate smoothly because they receive a value
          every tick, not just on state transitions.

Code    : yield return null;
Explain : Yields until the next frame. `yield return null` is the
          cheapest way to tick a coroutine once per frame — it is
          exactly equivalent to an Update method but keeps the state
          local.

DESIGN NOTES:

- Why route Fire()/StopFire() to StartCharge()/ReleaseCharge()?
  The framework's input code always calls Fire/StopFire. Routing to
  the charge API internally means the same input adapter works for
  bullet weapons, melee weapons, and this charge weapon with zero
  changes — pure polymorphism.

- Why clamp currentChargeTime instead of breaking out of the loop?
  Breaking out would end the coroutine and the listeners would stop
  receiving ChargeChangedEvents. Keeping the loop alive at 100%
  charge means the charge bar stays full and any pulsing "ready"
  VFX continues to fire.

- Why reset via RaiseChargeChanged(0f) in OnDisable AND ReleaseCharge?
  The charge bar needs to snap to 0 whenever the logical charge is
  gone, regardless of the reason. OnDisable handles "weapon turned
  off mid-charge" and ReleaseCharge handles "shot fired normally".
  Both paths publish the same event so the UI behaves identically.
