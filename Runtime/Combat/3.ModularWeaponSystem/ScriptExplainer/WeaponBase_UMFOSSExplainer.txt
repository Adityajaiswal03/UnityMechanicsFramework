PURPOSE:
The abstract base class for every weapon in the framework. Owns the
identity (name, skin, model) and shared stats (damage, range, weight),
and defines the equip/unequip lifecycle that publishes through the
WeaponEventBus. Capabilities like firing, reloading, and charging are
added by subclasses via interfaces — never by inheriting more methods
from this base. This composition-over-inheritance split is what lets
bullet weapons, melee weapons, and charged weapons share a single base
class without polluting each other's API.

Code    : public virtual void OnUnequip()

DESIGN NOTES:

- Why is this base class abstract?
  Dropping WeaponBase_UMFOSS directly on a GameObject would give you a
  weapon with no fire, swing, or charge behaviour. Making the class
  abstract turns that mistake into a compile-time error.

- Why composition (interfaces) instead of baking Fire/Reload into the
  base?
  Not every weapon needs every capability. A beam weapon does not
  reload; a melee weapon does not have ammo. Baking every capability
  into the base would force every weapon to carry dead methods. The
  capability interfaces (IWeaponFirable, IWeaponReloadable,
  IWeaponChargeable) let each concrete weapon declare exactly what it
  supports, and input/UI code uses `as` casts to query each capability.

- Why are all the fields private with protected accessors?
  Protecting the fields prevents subclasses from mutating base state in
  ways the base class did not anticipate. Reads are safe, writes are
  not — the base class owns the lifecycle of its own data.

- Why must base.OnEquip() be called LAST in overrides?
  The event fires from base.OnEquip(). If a subclass mutates state
  (initializes counters, resets timers) AFTER the base call, subscribers
  that read the weapon state on the event would see stale values.
  Calling base last guarantees the event reflects the fully-set-up
  weapon.
